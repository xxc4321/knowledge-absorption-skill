# Knowledge Absorption - 知识吸收优化器

## 核心理念

**大多数人混淆了两种知识吸收目标：**

| 目标 | 学习深度 | 所需时间 | 核心产出 | 适用场景 |
|-----|---------|---------|---------|---------|
| 学习级吸收 | 能解释原理、动手实现、排查问题 | 几十到几百小时 | 技能 | 你要成为这个领域的从业者 |
| 决策级吸收 | 能区分优劣、匹配场景、做出选择 | 15-60 分钟 | 判断 | 你要使用这个领域的工具或服务 |

**本Skill的核心价值**：为"决策级吸收"设计，帮助用户在15-30分钟内从"完全不懂"到"能做出正确选择"。

---

## 触发条件

### 自动触发场景

**"听不懂"类**（触发解释重构）：
- "听不懂"、"不明白"、"这说的什么"
- "能换个说法吗"、"太抽象了"、"不理解"
- "没听懂"、"懵了"、"啥意思"

**"做决策"类**（触发决策级吸收流程）：
- "XX和XX有什么区别"、"XX对比XX"
- "哪个更适合我"、"我该选哪个"
- "XX好不好"、"怎么选择"
- 或其他表达选择困惑的表述

### 混合场景处理

如果用户同时表达"听不懂"+"对比需求"（如"听不懂RAG和Wiki的区别"）：
1. **优先执行解释重构**（先用类比让用户理解两个概念）
2. **再执行决策级流程**（帮用户做出选择）

### 主动询问机制

如果无法判断用户意图，直接询问：
> "你问这个的主要目的是什么？是想快速做出选择（比如对比几款产品后决定用哪个），还是想深入理解原理？"

## 初始化流程（冷启动）

**触发条件**：首次使用时，检测到 `./USER.md` 没有"认知画像"模块

**引导话术**：
> "在我帮你更好地吸收新知识之前，我需要先了解你的学习习惯。回答几个简单问题就行："

**初始化问题**：

1. **思维偏好**：你觉得你更偏向文科思维（喜欢语言、故事、概念关联）还是理科思维（喜欢逻辑推理、因果关系）？
   - 选项：A. 文科 / B. 理科 / C. 混合型

2. **学习背景**：高中时学得最轻松/最拿手的是什么？（比如：语文、数学、物理、历史、英语...）

3. **当前熟悉领域**：你现在最熟悉的领域是什么？（比如：零售运营、飞书工具、B站创作...）

4. **解释偏好**：当你学习新东西时，哪种解释方式最容易让你"哦原来是这样"？
   - 选项：A. 具体案例 / B. 抽象逻辑 / C. 可视化（图表/视频）/ D. 动手操作

5. **最近"听不懂"的例子**：你最近一次觉得"这说的啥"的知识是什么？是卡在哪里？

**存储位置**：使用 `read_file` 工具读取 `./USER.md`，使用 `edit_file` 工具追加"认知画像"模块，格式如下：

```markdown
## 认知画像（YYYY-MM-DD初始化）

### 思维偏好
- **类型**：[文科/理科/混合型]
- **特点**：[根据用户回答总结]

### 学习背景
- **高中强项**：[科目]
- **高中弱项**：[科目]
- **大学专业**：[专业]

### 领域熟悉度（known/unknown）

#### ✅ 已掌握（known）
- [领域1]
- [领域2]

#### 🔄 正在学习
- [领域]

#### ❓ 待理解（unknown）
- [概念1]
- [概念2]

### 解释偏好
- **最有效方式**：[案例/逻辑/可视化/动手操作]
- **最佳类比来源**：[用户最熟悉的领域]

### 成功类比库
- [记录哪些类比用户一听就懂，方便复用]

### 类比失败记录
- [记录哪些类比用户听不懂，方便下次换一个]
```

## 两种流程对比

| 维度 | 决策级吸收流程 | 解释重构流程 |
|-----|--------------|-------------|
| 触发条件 | "XX和XX对比"、"选哪个" | "听不懂"、"不明白" |
| 用户目标 | 做出选择 | 理解概念 |
| 输出产物 | 决策矩阵+推荐结论 | 四层解释+验证 |
| 核心方法 | 维度对比+加权计算 | 类比翻译+层级展开 |
| 适用场景 | 工具选型、方案对比 | 新概念学习 |

## 执行流程

### 流程选择逻辑

```
用户提问
    │
    ├─ 包含"对比/选择"关键词？
    │   ├─ 是 + 用户已理解概念 → 执行【决策级吸收流程】
    │   └─ 是 + 用户不理解概念 → 先执行【解释重构】，再执行【决策级吸收】
    │
    ├─ 包含"听不懂"关键词？
    │   └─ 是 → 执行【解释重构流程】
    │
    └─ 无法判断 → 主动询问后选择流程
```

## 流程一：决策级吸收流程

### 快捷模式（适用于二选一场景）

当用户只需要在两个选项中选择时：
1. 直接对比两个选项的3个核心维度
2. 给出推荐 + 一句话理由

**示例**：
```
用户："Notion和Obsidian选哪个？"
AI："如果你更看重协作和AI功能，选Notion；如果你更看重数据安全和本地优先，选Obsidian。
    
    | 维度 | Notion | Obsidian |
    |-----|--------|----------|
    | 协作能力 | ⭐⭐⭐⭐⭐ | ⭐⭐ |
    | 数据安全 | ⭐⭐ | ⭐⭐⭐⭐⭐ |
    | AI功能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |"
```

### 完整模式（15分钟法）

**适用场景**：用户需要快速了解陌生领域并做出选择（3个以上选项）

---

#### Step 1：问题定义与场景采集（3分钟）

**操作**：帮助用户明确需要回答的问题、暂时不需要知道的问题，以及用户的具体场景。

**标准模板**：
```
我需要回答的问题：
1. X 是什么？它能解决我的什么问题？
2. X 有哪几种主流方案？
3. 对于我的 [具体场景]，哪种方案最合适？

我暂时不需要知道的问题：
- X 的底层技术原理
- X 的历史发展
- 如何从零搭建 X
```

**场景采集（必须完成）**：
询问用户以下信息（如果用户画像中已有可跳过）：
- **使用场景**：个人/团队/商业？
- **技能水平**：小白/有一定基础/技术人员？
- **核心诉求**：省时间/省成本/数据安全/功能强大？（让用户排序）

**为什么这一步最重要**：直接砍掉90%对用户无用的信息，同时确保后续推荐基于真实场景。

---

#### Step 2：框架扫描与分类提取（5分钟）

**操作**：搜索领域分类、主流方案、对比信息，快速获取整体框架。

**搜索关键词模板**：
```
"[领域名称] 分类"
"[领域名称] 主流方案"
"[领域名称] 对比"
"[领域名称] 怎么选择"
```

**只需提取**：
1. 这个领域解决的核心问题是什么？
2. 目前有哪3-5种主流方案？
3. 每种方案的一句话核心特点是什么？

**注意**：不要深入任何一个方案的细节，只需要知道有哪些分类和各自的核心定位。

---

#### Step 3：类比翻译与概念降维（3分钟）

**操作**：为每个核心概念找到用户熟悉的日常类比。

**标准模板**：
```
[专业术语] 就像是 [日常事物]，它的作用是 [一句话解释]。
```

**类比来源优先级**：
1. 用户known领域的场景（从 `./USER.md` 读取）
2. 用户成功类比库（优先使用历史成功的类比）
3. 日常生活中的常见事物
4. 通用技术类比模板（见附录）

**如果类比失败**：记录到 `./USER.md` 的"类比失败记录"，下次换一个。

---

#### Step 4：决策维度提取与权重分配（2分钟）

**操作**：根据用户的具体场景，提取3-5个最重要的决策维度，并分配权重。

**通用决策维度清单**（根据用户情况选择）：
- 上手难度/学习成本
- 部署难度/技术要求
- 数据安全性/隐私性
- 功能丰富度
- 性能/速度
- 成本/价格
- 生态/插件支持
- 可扩展性
- 离线可用性
- AI对话能力

**权重分配**：
让用户对每个维度分配权重（总和100%），示例：
```
- 上手难度：40%
- 数据安全性：30%
- AI对话能力：20%
- 成本：10%
```

如果用户不愿分配权重，默认采用等权重。

---

#### Step 5：决策矩阵构建与结论得出（2分钟）

**操作**：整理成矩阵，按评分标准打分（1-5分），加权计算总分，给出结论。

**评分标准（必须参照）**：

| 维度 | 1分 | 3分 | 5分 |
|-----|-----|-----|-----|
| 上手难度 | 需要编程+服务器部署 | 有图形界面但需配置 | 开箱即用 |
| 数据安全性 | 数据存储在第三方云端 | 本地存储但需联网 | 完全离线可用 |
| 成本 | 每月$50+ | 每月$10-50 | 免费 |
| 功能丰富度 | 功能单一 | 基础功能齐全 | 功能强大+插件生态 |
| 性能/速度 | 明显卡顿 | 偶尔延迟 | 流畅快速 |
| AI对话能力 | 无AI功能 | 基础问答 | 深度对话+上下文记忆 |
| 生态/插件 | 无插件 | 官方插件库 | 开放插件生态 |
| 可扩展性 | 功能固定 | 可配置 | API开放+自定义开发 |

**输出格式**：
```
| 方案 | 上手难度 | 数据安全 | AI能力 | 成本 | 加权总分 |
|-----|---------|---------|--------|-----|---------|
| 方案A | 4 | 3 | 5 | 4 | 3.9 |
| 方案B | 3 | 5 | 4 | 3 | 4.1 |

权重分布：上手难度40%，数据安全30%，AI能力20%，成本10%

结论：对于你的场景（个人使用+小白+省时间优先），推荐方案B，因为...
```

## 流程二：解释重构流程（4层结构）

**适用场景**：用户想深入理解某个概念，但当前解释听不懂

---

### 第一步：概念层级检测

**操作**：检查目标概念是否包含用户unknown的子概念。

**子概念拆解方法**：

**常见概念的子概念清单**（供AI参考）：

| 概念 | 子概念 |
|-----|--------|
| RAG | Embedding、向量数据库、检索、生成 |
| LLM | Transformer、预训练、微调、推理 |
| 知识库 | 存储、检索、组织结构、权限管理 |
| Agent | 规划、工具调用、记忆、执行 |
| 向量数据库 | 向量、索引、相似度计算 |
| 微调 | 预训练模型、数据集、训练参数 |

**动态拆解方法**：
如果不在清单中，让AI自问：
> "要理解X，用户必须先理解哪些更基础的概念？"

**检测逻辑**：
```
目标概念 → 拆解子概念 → 检查用户known/unknown列表
    │
    ├─ 所有子概念都在known → 直接解释目标概念
    │
    └─ 存在子概念在unknown → 先解释子概念（洋葱剥皮法）
```

**洋葱剥皮法**：
- 第一层：解释最核心的概念（不依赖其他概念）
- 第二层：基于第一层，解释进阶概念
- 第三层：基于前两层，解释最终目标概念

---

### 第二步：读取用户认知画像

**操作**：使用 `read_file` 工具读取 `./USER.md` 中的"认知画像"模块

**读取内容**：
1. 思维偏好（文科/理科/混合型）
2. 学习背景（擅长/不擅长的学科）
3. 领域熟悉度（known/unknown）
4. 解释偏好（案例/逻辑/可视化/动手操作）
5. 最佳类比来源
6. 成功类比库（优先使用）

---

### 第三步：诊断上一次解释的问题

**操作**：检查是否存在以下问题

| 问题类型 | 检查方法 |
|---------|---------|
| 术语堆砌 | 是否用专业词汇解释专业词汇？ |
| 类比缺失 | 是否没有用户熟悉的参照物？ |
| 层级跳跃 | 是否从0直接到100，没有中间过渡？ |
| 使用unknown概念 | 是否用了用户unknown领域的概念来解释？ |

---

### 第四步：四层结构重构输出

**第一层：生活化类比**（用用户熟悉的领域）
- 从用户的known列表中选择最相关的领域
- 优先使用"成功类比库"中的类比
- 用该领域的场景/概念做类比

**第二层：核心概念的本质**（剥离术语后是什么）
- 用最朴素的语言说明这个概念在解决什么问题
- 不要出现任何专业术语

**第三层：专业术语对照**（现在你知道它为什么叫这个名字）
- 解释术语的字面含义
- 说明为什么叫这个名

**第四层：实际应用场景**（这东西能干嘛）
- 结合用户的实际需求/场景
- 给出具体的使用建议

---

### 第五步：快速验证与多轮优化

**验证环节**：
解释完成后，进行快速验证：
> "用你自己的话说一遍这个概念是干嘛的，我看看你是不是真懂了"

**验证判断标准**：

| 判断 | 特征 | 示例（问：什么是RAG？） |
|-----|------|----------------------|
| ✅ 正确 | 提到核心作用 + 用自己的话 + 能举场景 | "就是让AI先查资料再回答，不会瞎编" |
| ❌ 错误 | 只复述术语 | "检索增强生成" |
| ❌ 错误 | 太模糊，没说核心机制 | "就是让AI变聪明" |
| ❌ 错误 | 完全偏离 | "就是数据库" |

**多轮优化机制**（最多3轮）：

| 轮次 | 失败后策略 |
|-----|----------|
| 第1次失败 | 换类比角度（从用户known列表中换一个领域，或使用成功类比库中的替代类比） |
| 第2次失败 | 换解释偏好（案例→逻辑→可视化→动手操作） |
| 第3次失败 | 触发人工介入 |

**人工介入话术**：
> "我试了3种解释方式，好像都没讲清楚。要不你告诉我，你觉得最难理解的是哪个部分？我们一起拆开来看。"

**更新认知状态**：
- 如果用户回答正确 → 使用 `edit_file` 更新 `./USER.md`，将该概念从unknown转移到known，并将成功的类比记录到"成功类比库"
- 如果用户仍然不懂 → 记录失败的类比到"类比失败记录"

## 为什么听不懂？——底层原因分析与修复方案

当用户表示"听不懂"时，告知用户真实原因，并应用对应的修复方案：

| 原因 | 表现 | 占比 | 修复方案 |
|-----|------|-----|---------|
| 缺少"问题上下文" | 知道了"是什么"，但不知道"为什么要有这个东西" | 45% | 先说明"这个概念解决了什么痛点"，再解释概念 |
| 专有名词的"知识压缩"效应 | 每个术语背后都是一整套知识体系，缺任何一环都听不懂 | 30% | 每个术语单独解释，或用类比替代术语 |
| 认知过载 | 一次性给超过7±2个新概念，超过工作记忆容量 | 20% | 一次只讲1-2个新概念，分多次输出 |
| 知识储备不足 | 完全没有相关领域的基础图式 | 5% | 触发初始化，从用户known领域切入 |

**关键结论**：你听不懂95%的新技术，不是因为你笨，也不是因为知识储备不够，而是因为讲解者用了错误的顺序和粒度。

## 附录一：常见技术类比模板

**数据存储类**：
- 数据库 → 仓库
- 向量数据库 → 图书馆的智能索引系统（能理解内容相似性）
- 缓存 → 前台货架（常用的东西放前面）

**AI类**：
- LLM（大语言模型） → 过目不忘的店员（只懂培训时学过的）
- RAG → 聪明店员+实时库存系统（能查实时数据）
- Embedding → 给文字拍"语义照片"（意思相近的照片看起来也相似）
- Chunking → 把厚书拆成卡片（方便快速找到需要的那一页）
- Fine-tuning → 给店员做专项培训（让他更懂你的业务）
- Agent → 会自己干活的店员（不只是回答问题，还会执行任务）

**网络类**：
- 服务器 → 餐厅后厨
- API → 服务员（传递请求和响应）
- 缓存 → 前台货架

**通用类**：
- 算法 → 菜谱
- 索引 → 目录
- 协议 → 通用语言/规则

## 附录二：完整示例

**用户问题**："我想搭建个人知识库，不知道选哪种方式"

### Step 1：问题定义与场景采集

**AI输出**：
```
我来帮你选。先确认几个问题：

你需要回答的问题：
1. 个人知识库有哪几种主流方式？
2. 对于你的场景，哪种最合适？

我暂时不需要你深入了解的：
- 向量数据库的索引算法
- RAG的Chunking策略
- 如何自己训练模型

你的场景是？
- 使用场景：个人/团队？
- 技术水平：小白/有基础/技术人员？
- 最看重的：省时间/省成本/数据安全/功能强大？
```

**用户回答**："个人用，小白，想省时间，数据也要安全"

### Step 2：框架扫描与分类提取

**AI输出**：
```
个人知识库的主流方案：

1. 传统笔记软件（Notion、Obsidian）
```
